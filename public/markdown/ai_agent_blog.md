# 从聊天框到工作流：AI Agent 工程化使用指南

>【注意！】本文整理自个人 AI 工具观察、Codex 使用经验和本地知识库实践。AI 工具迭代很快，文中涉及的产品界面和功能以 2026-05-26 前后的状态为参考，后续可能变化。请把本文当作一份方法论和入门教程，而不是固定不变的软件说明书。

过去几年，很多人已经习惯了 ChatGPT、Claude 这类聊天式 AI：把问题贴进去，等模型回答，再把答案复制回自己的项目。这个模式适合解释概念、润色文字、生成片段，但一旦任务涉及多个文件、命令行、测试、日志、浏览器预览和反复修改，人就会变成“信息搬运工”。

AI Agent 改变的正是这件事。

它的价值不只是模型回答得更好，而是它能在真实工作现场闭环执行：读取项目上下文，调用工具，修改文件，运行验证，把结果留在文件系统和 Git diff 里。对工程师、研究者、产品经理和知识工作者来说，这意味着 AI 不再只是问答窗口，而可以参与一段完整的工作流。

这篇文章讲三件事：

- AI Agent 和普通 ChatBot 的本质区别。
- 哪些任务适合交给 Agent，哪些不适合。
- 如何用工程化方法配置、约束、验证和长期使用 Agent。

---

## 1. 从 ChatBot 到 Agent：先理解范式变化

你可能已经熟悉 ChatGPT 网页版或 Claude 网页版。它们的主要形态是 ChatBot：你在浏览器中输入问题，它在对话框中回答。如果你想让它分析代码、数据、文档或报错日志，通常需要手动复制粘贴。

Agent 的工作方式不同。以 Codex、Claude Code、Kilo Code、OpenCode 等工具为例，Agent 通常运行在你的项目目录、本地工作区或 IDE 环境中，可以直接读文件、执行命令、编辑文件、运行测试，并根据结果继续行动。

### 1.1 ChatBot 和 Agent 的区别

| 方面 | ChatBot | Agent |
| --- | --- | --- |
| 典型入口 | 网页聊天窗口 | 终端、IDE、桌面应用、本地工作区 |
| 文件访问 | 需要手动上传或复制粘贴 | 可以直接读取工作区文件 |
| 命令执行 | 通常不能执行本地命令 | 可以运行 shell、测试、脚本、构建命令 |
| 文件修改 | 通常只能给出建议 | 可以直接编辑代码、文档、配置 |
| 上下文来源 | 主要来自你贴进去的内容 | 来自项目目录、工具结果、历史线程 |
| 工作流 | 人在多个窗口间搬运信息 | Agent 在工作区内闭环执行 |

Agent 的核心优势不是“模型一定更聪明”，而是它可以直接与真实工作环境交互。它能自己读取上下文、执行验证、观察结果、修改文件，再继续下一步。

### 1.2 Agent 更像控制系统

从闭环控制的角度看，ChatBot 和 Agent 都仍然是 human-in-the-loop，但人的位置不同。

在 ChatBot 工作流中，人经常充当“传感器”和“执行器”：复制文件内容、粘贴错误日志、运行命令、把回答写回项目。

在 Agent 工作流中，人更像“参考信号”和“裁决者”：设定目标、给出约束、审查结果、决定是否接受变更。读取、执行、编辑和验证这些中间步骤，更多由 Agent 完成。

一个典型 Agent 闭环可以写成：

```text
目标 -> 读取上下文 -> 制定计划 -> 执行动作 -> 观察结果 -> 修正动作 -> 交付结果
```

这个转变非常关键：你不再只是问 AI 一个问题，而是在给一个本地执行者分配任务。

### 1.3 基模、产品和工作形态不要混淆

讨论 AI 工具时要区分三层：

- 基础模型：GPT、Claude、Gemini 等模型家族。
- 产品形态：ChatGPT、Claude、Codex、Kilo Code、OpenCode 等面向用户的工具。
- 工作形态：ChatBot、Agent、IDE 插件、自动化任务、图像生成器等。

同一个基础模型可以被做成聊天产品，也可以被做成 Agent 产品。使用体验的差别往往不只来自模型能力，还来自它能接触什么上下文、能调用什么工具、能否留下可审查的文件变更。

## 2. Agent 不只是编程工具

很多人第一次接触 Agent 是因为写代码，但 Agent 的优势并不局限于编程。只要任务同时满足下面几个条件，就很适合交给 Agent 协助：

- 上下文分散在多个本地文件中。
- 需要反复执行命令、脚本、检查或构建。
- 需要把结果写回文档、代码、表格、配置或图像。
- 需要边做边检查，而不是一次性回答。
- 人类需要保留最终判断权。

工程、文档、知识管理和数据处理都大量符合这些条件。

### 2.1 接手陌生代码库

接手一个项目时，Agent 的第一价值不是立刻改代码，而是帮你建立地图。

可以这样问：

```text
请阅读当前项目，生成一份项目地图。
包括：
1. 主要目录和模块；
2. 启动入口；
3. 测试入口；
4. 配置文件；
5. 核心业务流程；
6. 修改时最需要小心的地方。
不要修改文件。
```

这类任务非常适合 Agent，因为它可以实际查看文件结构，而不是只根据你贴出来的几段代码猜测。

### 2.2 排查可复现的 bug

Agent 很适合处理有日志、有复现步骤、有测试入口的问题。

示例：

```text
请排查用户导入 CSV 时偶尔报 500 的问题。
已知复现步骤在 docs/repro-csv-import.md。
要求：
- 先定位导入流程相关代码；
- 找到最可能的异常路径；
- 优先补一个能复现问题的测试；
- 再做最小修复；
- 运行相关测试；
- 不要重构无关模块。
```

这里的重点是“可复现”和“可验证”。Agent 可以读日志、写测试、跑测试、看失败，再继续修。

### 2.3 补测试、补文档、补类型

很多工程任务并不难，但很碎：补测试、更新 README、调整类型定义、同步 API 文档、整理 changelog。Agent 对这类任务很有价值，因为它能处理跨文件一致性。

示例：

```text
请为订单导出接口补充分页参数说明。
要求：
- 阅读现有路由、测试和 OpenAPI 文档；
- 不改变接口行为；
- 只补充文档和必要的测试断言；
- 最后运行相关测试并给出 diff 摘要。
```

这种任务交给 ChatBot 往往会停留在“建议怎么写”；交给 Agent 则可以直接落到文件变更。

### 2.4 自动化重复性资料处理

Agent 可以把自然语言意图翻译成脚本、命令和报告。适合自动化的数据处理任务包括：

- 批量读取 CSV、JSON、日志或 Markdown。
- 转换格式。
- 汇总结果。
- 生成图表。
- 写入 Markdown、HTML、PPT 或表格。
- 保存可复用脚本。

示例：

```text
请读取 logs/ 目录下最近 7 天的应用日志。
统计每类错误码的出现次数和首次出现时间。
输出一份 Markdown 报告到 tmp/error-summary.md。
如果需要写脚本，请放在 scripts/analyze_logs.py。
运行后保留脚本，方便下次复用。
```

关键不是让 Agent 凭空“总结趋势”，而是让它在你的真实文件上执行，并把过程留下来。

### 2.5 前端页面和交互验收

Agent 如果具备浏览器能力，就不应该只靠读代码判断 UI。

示例：

```text
请检查设置页在移动端是否有按钮溢出。
要求：
- 启动本地开发服务；
- 用浏览器打开设置页；
- 检查 375px 和 1280px 两个宽度；
- 如果发现问题，定位相关组件和样式；
- 修改后再次用浏览器验证；
- 最后说明验证结论。
```

前端问题往往出现在真实渲染、窗口尺寸、字体、滚动和交互状态里。Agent 能打开页面时，就应该让它看实际结果。

## 3. 一个完整工程流程示例

下面用一个通俗的工程任务串起来：修复“用户导入 CSV 时空行导致报错”的问题。

这个例子不依赖具体产品。无论你使用 Codex、Claude Code、Kilo Code 还是 OpenCode，都可以按类似思路工作。

### 第一步：先建立项目地图

```text
请阅读当前项目，找到用户 CSV 导入功能的实现位置。
请输出：
1. 相关路由或命令入口；
2. CSV 解析逻辑所在文件；
3. 现有测试位置；
4. 错误处理和日志记录方式；
5. 你建议重点检查的代码路径。
先不要修改文件。
```

目标是让 Agent 先形成整体认识：入口在哪里，解析在哪里，测试在哪里，错误如何返回。

### 第二步：把问题变成可复现测试

```text
请为 CSV 导入增加一个失败复现测试。
场景：
- CSV 文件中间包含一个完全空行；
- 合法数据行仍然应该被导入；
- 空行应该被跳过；
- 错误行仍然要返回准确行号。
先只写测试并运行，确认当前实现会失败。
```

这一步很重要。没有复现测试，Agent 很容易做出“看起来合理”的改动，却没有证据证明它解决了问题。

### 第三步：做最小修复

```text
现在请修复这个问题。
要求：
- 只修改 CSV 导入相关代码；
- 不改变合法 CSV 的现有行为；
- 不引入新依赖；
- 保持错误行号和原始文件行号一致；
- 修复后运行刚才新增的测试和相关导入测试。
```

这一步的重点是边界清楚。你告诉 Agent 可以改哪里、不能改什么、如何验证。

### 第四步：扩大验证范围

```text
请再检查以下边界情况是否已有测试覆盖：
1. 文件末尾空行；
2. 只有表头没有数据；
3. 带 BOM 的 UTF-8 文件；
4. 某一行字段数量不足；
5. 某一行字段类型错误。
只补必要测试，不做无关重构。
```

很多真实 bug 都来自边界条件。Agent 可以帮你系统扫描，但仍然要把范围限制在当前问题附近。

### 第五步：把经验写入文档

```text
请把这次修复补充到 docs/runbook.md。
要求：
- 记录问题现象；
- 记录根因；
- 记录复现方式；
- 记录修复策略；
- 记录以后排查 CSV 导入问题时的检查顺序。
```

这一步会把一次临时修复变成团队资产。下次遇到类似问题，Agent 和人都能直接读取稳定文档，而不是翻聊天记录。

这个流程的核心不是“让 Agent 自己决定一切”，而是你持续提供目标、边界和验收标准。Agent 负责执行中间劳动，你负责方向和质量。

## 4. 选择和配置 Agent 工具

不同工具的界面和生态会变化，但一个实用 Agent 环境通常需要具备几类能力。

| 能力 | 作用 | 使用建议 |
| --- | --- | --- |
| 文件读写 | 修改代码、文档、配置、数据文件 | 明确允许范围，避免无边界批量改动 |
| Shell 命令 | 测试、构建、脚本、格式化、数据处理 | 把验证命令写进任务描述 |
| 浏览器 | 本地网页预览、表单测试、截图验收 | 前端任务要看真实页面 |
| 桌面操作 | 处理只能通过 GUI 完成的流程 | 涉及发送、删除、提交前应确认 |
| 连接器 | Issue、PR、邮件、日历、消息等外部上下文 | 只给完成任务所需的最小权限 |
| 自动化 | 定期检查、提醒、监控、复盘 | 明确触发条件、权限和输出方式 |

工具名称并不是关键。关键是你的工作区是否可操作、可验证、可审查。

### 4.1 先从本地工作区开始

第一次使用 Agent，建议选择一个边界清楚的本地目录，例如：

```text
my-project/
  README.md
  docs/
  src/
  tests/
  tmp/
```

不要一上来就把 Agent 放进包含大量私人资料、生产凭据和无关仓库的目录。工作区越清楚，Agent 越容易稳定执行。

### 4.2 写一份给 Agent 的规则文件

不同产品对规则文件的命名不完全一样。有的使用 `AGENTS.md`，有的使用 `CLAUDE.md`，有的读取项目 README、配置文件或自定义说明。名字不关键，关键是要有一份写给 Agent 的操作契约。

一份有效规则文件应该说明：

- 这个项目是做什么的。
- 主要目录分别存放什么。
- 如何安装、启动、测试、构建。
- Python、Node、Rust、Go 等环境应该怎么使用。
- 哪些文件不能改，哪些目录可以放临时产物。
- 修改代码后必须运行哪些验证。
- 是否需要创建 Git commit，以及 commit message 规范。
- 对安全操作、依赖变更、批量重命名的限制。

一个简化版本可以这样写：

```md
# Agent 工作规则

## 项目说明
这是一个内部订单管理服务，后端使用 FastAPI，前端使用 React。

## 常用命令
- 启动后端：`uv run python -m app`
- 运行测试：`uv run pytest`
- 前端检查：`npm run lint && npm run test`

## 修改规则
- 不要直接修改生产数据库迁移文件，除非任务明确要求。
- 临时分析文件放入 `tmp/`。
- 修改接口行为时，同步更新测试和 API 文档。
- 完成后必须给出 diff 摘要和验证结果。
```

规则文件的意义不是“让 AI 变听话”，而是把你的工程习惯写成可执行上下文。Agent 看到明确规则后，才更容易稳定地做出符合团队预期的动作。

### 4.3 第一个任务要小而完整

不要一上来就让 Agent “重构整个项目”或“整理全部资料”。更好的第一个任务是：

```text
请检查 README.md 中的本地启动命令是否和当前项目一致。
要求：
- 只修改 README.md；
- 如果命令不确定，先查看 package.json 和 pyproject.toml；
- 不安装新依赖；
- 最后给出改动摘要和未验证项。
```

你需要通过这种小任务观察：

- 它是否会先读文件。
- 它是否能遵守边界。
- 它是否会运行或说明验证。
- 它是否会清楚报告改动。
- 它是否尊重已有未提交变更。

### 4.4 权限越大，规则越要清楚

Agent 能力越强，越需要清楚的权限边界。尤其是这些操作，不要默认授权：

- 删除或批量移动文件。
- 修改依赖和锁文件。
- 修改数据库迁移。
- 格式化整个仓库。
- 自动发送邮件、消息、表单。
- 访问包含敏感信息的外部系统。

Agent 很适合执行复杂操作，但前提是范围、理由、验证和回滚方式都清楚。

## 5. 从短聊天到长期工作流

Agent 的潜力不只在于单次回答，而在于把工作变成可以持续推进的循环。下面这些能力和使用习惯，决定了它能不能从“问答工具”升级为“工作环境”。

### 5.1 持久线程：让上下文有连续性

长期线程适合承载一个稳定工作流，例如：

- 某个软件仓库。
- 某个产品功能。
- 某篇文档或报告。
- 某套知识库。
- 某个长期排查任务。
- 周期性信息监控。

长期线程的优点是连续性强，缺点是上下文漂移风险更高。

建议：

- 重要工作流使用固定线程。
- 关键决策写入文件，而不是只留在聊天记录里。
- 定期让 Agent 总结当前状态到 `TODO.md`、项目日志或决策记录。
- 不要把完全无关的任务塞进同一个长期线程。

### 5.2 Steering：边工作边追加方向

Steering 指的是在 Agent 已经工作时继续补充方向，而不是等它完全结束再说下一句。

例如它正在改一个设置页，你可以继续追加：

```text
这个按钮再紧凑一点。
标题不要用营销文案，改成操作型文案。
完成后打开本地预览。
如果测试通过，再准备 commit。
```

这种工作方式更接近你和同事协作：你不需要把所有要求一次性想完，可以在看到中间结果后继续修正方向。

### 5.3 Memory：重要信息要落盘

长期线程本身不是可靠的知识库。真正可靠的记忆应该写入文件，变成可以检查、编辑、对比和复用的内容。

推荐在项目中维护这些文件：

```text
docs/
  architecture.md      # 架构说明
  runbook.md           # 运维和排查手册
  decisions.md         # 关键决策记录
  changelog.md         # 重要变更摘要
tmp/
  investigation-notes/ # 临时分析材料
```

你可以这样要求 Agent：

```text
这次问题排查结束后，请把结论写入 docs/runbook.md。
要求：
- 记录故障现象；
- 记录根因；
- 记录排查命令；
- 记录修复方式；
- 记录下次遇到相似问题时的检查顺序。
```

这样做有三个好处：

- 新成员能直接读文档，不必翻聊天记录。
- Agent 下次也能读取稳定文件，而不是依赖模糊记忆。
- Git diff 能审查它到底“记住”了什么。

### 5.4 Browser、Computer 和连接器

可以把不同工具的职责这样区分：

| 工具类型 | 适合场景 |
| --- | --- |
| 内置浏览器 | 本地网页预览、静态 HTML、Storybook、Streamlit、管理后台等 |
| 系统浏览器或已登录浏览器 | 需要现有登录态、多标签、真实网页交互 |
| 桌面操作 | 只能通过本地 GUI 完成的任务 |
| Issue/PR/邮件/日历连接器 | 工作消息、评审、排期、外部上下文 |

原则是：让 Agent 接触完成任务所需的最小界面，不要一开始就给它无边界地操作所有东西。

### 5.5 自动化：让线程定期醒来

很多 Agent 产品正在支持定时任务、提醒、监控或线程唤醒。这类能力适合做：

- 定期检查 PR 评论。
- 定期查看 Issue 状态变化。
- 定期更新项目状态摘要。
- 定期检查某个文档是否有新评论。
- 在等待外部事件时继续监控。

示例：

```text
每 30 分钟检查一次这个 PR 是否有新评论。
如果有评论，请总结需要处理的问题，判断是否需要改代码。
不要自动 push，先把建议发在这个线程里。
```

自动化的关键不是“让 Agent 自动做一切”，而是让它在合适的时间重新收集上下文，把需要人判断的内容带回来。

### 5.6 侧面板和工件预览：让成果可审查

很多 Agent 工具会提供侧面板、预览区或工件视图。它可以承载：

- Markdown 文档。
- CSV 和表格。
- PDF。
- 幻灯片。
- 本地网页。
- 代码 diff。
- 浏览器交互状态。

这意味着你可以直接查看 Agent 正在生成或修改的对象，并在同一个界面里反馈。对知识工作和前端工程来说，这一点非常重要：成果不再只是一段回答，而是可以被审阅的工件。

## 6. 工程和知识工作的典型用法

下面这些场景展示了 Agent 参与实际工作的常见形态：它可以读材料、整理问题、生成脚本、修改文档、协助复盘，但最终判断仍然由人完成。

### 6.1 代码库接手与项目地图

适合交给 Agent 的任务包括：

- 扫描目录结构。
- 找到启动入口。
- 找到测试入口。
- 梳理核心业务流程。
- 标出高风险模块。
- 生成 onboarding 文档草稿。

示例：

```text
请为这个仓库生成一份新同事入门文档草稿。
要求：
- 先阅读 README、package.json、pyproject.toml 和 src/；
- 说明本地启动步骤；
- 说明测试和 lint 命令；
- 画出请求从路由到数据库的大致流程；
- 不要修改文件，先给草稿。
```

### 6.2 Bug 排查与复盘

适合交给 Agent 的任务包括：

- 阅读错误日志。
- 定位相关代码。
- 写复现测试。
- 做最小修复。
- 补充回归测试。
- 生成复盘文档。

示例：

```text
请根据 tmp/error.log 排查支付回调偶发重复入账的问题。
要求：
- 先定位回调处理链路；
- 判断是否存在幂等性缺口；
- 如果能复现，先写测试；
- 修复必须保持现有接口兼容；
- 最后把排查结论写入 docs/incidents.md。
```

这种任务的关键是：让 Agent 先找证据，再写代码。

### 6.3 文档和知识库维护

Agent 很适合维护工程化文档库。一个适合团队协作的文档库可以这样组织：

```text
knowledge-base/
  services/
    payment/
      overview.md
      api.md
      incidents.md
    order/
      overview.md
      api.md
      incidents.md
  decisions/
    2026-05-26-use-job-queue.md
  playbooks/
    rollback.md
    database-migration.md
  templates/
    decision-record.md
    incident-review.md
  tmp/
```

这里的重点不是目录名本身，而是规则清楚：

- 每个服务有固定入口文档。
- 决策记录按日期命名。
- 事故复盘有统一模板。
- 临时分析产物不混进正式文档。
- 大规模改名或移动前必须先给计划。

可以给 Agent 一个维护任务：

```text
请扫描 knowledge-base/services/order/ 下的文档。
目标是找出过期的接口描述和缺失的排查步骤。
要求：
- 只诊断，不修改；
- 每个问题都给出文件路径和依据；
- 区分“确定过期”和“需要人工确认”；
- 最后给出建议的修改顺序。
```

这比“帮我整理文档”更有效，因为它把 Agent 的工作限制在诊断、证据和计划上。

### 6.4 个人资料库和文献库

如果希望 Agent 长期维护资料库，目录设计要对 Agent 友好，也要对人可审查。

推荐原则：

- Markdown、YAML、JSON、BibTeX 等文本文件作为事实来源。
- 数据库和 GUI 只作为索引、缓存或展示层。
- 每个资料单元有独立文件夹。
- 附件放在该单元自己的 `attachments/` 中。
- 不确定内容必须标记，不能自动当事实写死。
- 批量移动、重命名、删除前必须先给计划。

一个简化资料单元可以这样设计：

```text
notes/
  2026-05-26-csv-import-runbook/
    note.md
    meta.yaml
    attachments/
```

`meta.yaml` 保存稳定身份、标题、标签、创建时间和摘要：

```yaml
id: "b4a9b1f2-1f2a-4d7a-9a1f-7c7d8a111111"
title: "CSV 导入问题排查记录"
category: ["工程", "数据导入"]
tags: ["runbook", "bugfix", "csv"]
created_at: "2026-05-26T10:00:00+08:00"
summary: "记录 CSV 导入空行导致报错的复现、修复和验证方式。"
```

Agent 可以参与这些维护任务：

- 新建符合规范的资料文件夹。
- 修复缺失 ID、分类格式错误、标签格式错误。
- 根据正文生成 summary。
- 发现相关笔记并提出双向引用建议。
- 生成主题索引笔记。
- 先提出分类迁移计划，再等待确认后批量修改。

Agent 不应该默认做这些事：

- 未确认就批量移动文件夹。
- 未确认就删除笔记或附件。
- 大规模重写用户正文。
- 把 GUI 缓存当作事实来源。

### 6.5 图像、截图和可视化

很多 Agent 工具可以配合图像生成、图表生成、Mermaid、网页截图或幻灯片。它们很适合提升表达效率，但要区分“解释性视觉”和“证据性视觉”。

适合生成或整理的内容：

- 架构图。
- 流程图。
- 教程配图。
- 信息图。
- 报告封面。
- UI 草图。
- 从真实数据生成的图表。

不适合混用的内容：

- 把生成图片伪装成真实实验结果。
- 用 AI 图像替代需要可追溯来源的产品图。
- 在严肃报告里使用无法解释来源的“看起来像数据”的图。
- 让读者误以为图像来自真实观测、仿真或生产系统。

更工程化的用法是：

```text
请把 docs/architecture.md 里的任务调度流程整理成一张 Mermaid 流程图。
要求：
- 只基于文档中已有信息；
- 不新增不存在的服务；
- 节点名称使用代码库里的真实模块名；
- 生成后把图插入文档对应位置。
```

或者：

```text
请根据当前前端页面生成一份 UI 验收报告。
要求：
- 用浏览器检查桌面和移动端宽度；
- 保存必要截图到 tmp/ui-check/；
- 报告里只描述真实截图观察到的问题；
- 不要使用生成图替代真实页面截图。
```

这里的图不是装饰，而是对已有系统或真实界面的结构化表达。

## 7. 写给初学者的 Agent 使用原则

### 7.1 先给边界，再给目标

坏例子：

```text
帮我整理这个项目。
```

好例子：

```text
请整理 docs/ 目录下的入门文档。
要求：
1. 只修改 docs/ 下的 Markdown；
2. 不移动图片和附件；
3. 先列出你发现的问题；
4. 我确认后再统一改写；
5. 修改后运行 markdown lint，如果项目没有 lint，就说明未运行。
```

边界越清楚，Agent 越不容易过度发挥。

### 7.2 让 Agent 先读，再改

对陌生项目，第一轮最好要求：

```text
先阅读项目结构和相关文件，给出你的理解和修改计划。
在我确认前不要改文件。
```

当你已经信任某个工作区的规则，并且任务很小，再让它直接执行。

### 7.3 把验证写进 prompt

不要只说“改好”。要说明什么叫好：

```text
修改完成后请运行：
npm run test -- user-import
npm run lint
git diff --check

如果有命令失败，请报告失败原因和下一步建议。
```

验证命令越具体，Agent 越容易闭环。

### 7.4 用 Git diff 审查 Agent 的记忆

如果让 Agent 维护长期记忆、知识库或项目状态，一定让它写文件，并通过 Git diff 审查。不要让重要信息只留在对话上下文中。

### 7.5 不要把高风险操作合并进大任务

下面这些操作应当单独确认：

- 批量移动文件。
- 删除文件。
- 重写 metadata。
- 改依赖。
- 改构建系统。
- 格式化整个仓库。
- 自动发送邮件或消息。

Agent 很适合执行这些操作，但前提是范围、理由和验证都清楚。

## 8. 常用 prompt 模板

### 8.1 项目地图

```text
请阅读当前项目，生成一份项目地图。
包括：
1. 主要目录和文件；
2. 启动方式；
3. 测试方式；
4. 核心业务流程；
5. 关键配置；
6. 修改时需要注意的风险。
不要修改文件。
```

### 8.2 安全修改

```text
请完成以下修改：<任务描述>。
约束：
- 只修改 <允许范围>；
- 不新增依赖；
- 不移动文件；
- 保留已有未提交改动；
- 修改后运行 <验证命令>；
- 最后给出 diff 摘要、验证结果和剩余风险。
```

### 8.3 Bug 排查

```text
请排查这个问题：<问题描述>。
已知复现步骤：<步骤>。
要求：
- 先定位相关代码路径；
- 解释可能根因；
- 优先写一个能复现问题的测试；
- 再做最小修复；
- 运行相关测试；
- 不要顺手重构无关代码。
```

### 8.4 前端验收

```text
请检查 <页面或功能> 的交互是否正常。
要求：
- 启动本地开发服务；
- 用浏览器打开页面；
- 测试核心路径：<路径列表>；
- 检查桌面和移动端宽度；
- 记录发现的问题；
- 只有在明确需要时才修改代码。
```

### 8.5 文档整理

```text
请整理 <目录> 下的文档。
要求：
- 先列出问题清单，不要直接修改；
- 区分事实错误、结构混乱、表达不清和信息缺失；
- 每条问题都给出文件路径；
- 我确认后再按优先级修改；
- 修改时保留原有技术结论。
```

### 8.6 长期任务

```text
请把这个任务作为长期目标推进：<目标>。
完成标准：
1. <可验证标准 A>；
2. <可验证标准 B>；
3. <可验证标准 C>。
每轮行动后更新 TODO.md，说明已完成、下一步和阻塞点。
```

### 8.7 资料库维护

```text
请扫描 notes/ 下的资料库，找出分类和标签的整理建议。
要求：
1. 只诊断，不修改；
2. 区分明显错误和主观整理建议；
3. 对每条建议给出受影响文件数量；
4. 批量修改前等待我确认。
```

## 9. 结语

Agent 的真正价值不是替代人的判断，而是把大量中间劳动搬进一个可执行、可检查、可迭代的工作环境中。

用好 Agent 的关键可以概括为五句话：

- 文件系统是事实来源。
- 规则文件是协作契约。
- 脚本和测试负责验证。
- Agent 负责读取、整合和执行。
- 人负责目标、边界、审查和最终判断。

当你按这个原则组织工作区后，Codex、Claude Code、Kilo Code、OpenCode 这类 Agent 就不再只是“能帮你写几段代码的聊天框”，而会变成一个可以持续参与工程、文档、知识管理和自动化任务的协作系统。
